Created by Kacy Tran, Anh Minh, and Bao Long.

Mate is a cross platform mobile application built with Flutter for managing a care and companionship service. It connects three groups of people around subscription based rooms: customers who buy a care package, staff members who look after those customers and record daily attendance, and administrators who oversee the whole operation. The app supports English and Vietnamese, real time chat, an AI powered assistant, and online payments.

## Project history

Mate was created in 2024 as a real product for a small care service business. The goal was to give families a simple way to subscribe to a care room, stay in touch with the staff assigned to their loved ones, and follow daily attendance and activity from their phone. The team shipped a working build across Android, iOS, web, and desktop, wired up Firebase for authentication support and media storage, and integrated PayPal so customers could pay for their packages directly in the app.

Later on, because of several aspects of the business and its management that were outside the control of the engineering team, the original service was shut down and the product was taken offline. Rather than let the work disappear, the codebase has been released as open source so that students, developers, and anyone interested in Flutter can learn from it, reuse the parts they find useful, and continue building on top of it. The project is shared as is, for educational and reference purposes.

## Features

### For customers
- Register and log in with email and password or with a Google account.
- Browse and subscribe to care packages, offered as Gold, Silver, and Bronze rooms.
- Pay for a subscription securely through PayPal inside the app.
- View the assigned room, the staff member in charge, and daily attendance history.
- Filter attendance by week, month, or year.
- Chat in real time with staff.
- Read news and notifications.
- Manage a personal profile, including avatar, address, language, and password.

### For staff
- Log in to a dedicated staff workspace.
- View the customers and rooms they are responsible for.
- Record and review attendance for each customer.
- Chat with customers and respond to requests.
- Edit their own staff profile.

### For administrators
- Access an admin dashboard with an overview of customers and staff.
- See analytics such as active customer and staff counts and monthly cash flow.
- Manage staff, customers, rooms, and events.
- Monitor messages and notifications across the service.

## Technology stack

- Flutter and Dart for the application across Android, iOS, web, Linux, macOS, and Windows.
- flutter_bloc and bloc for state management, following the BLoC pattern.
- Firebase Core, Firebase Realtime Database, and Firebase Storage for realtime chat and media.
- A REST backend accessed through the http package.
- PayPal integration for payments through the flutter_paypal package.
- Google Sign In for social authentication.
- Google Generative AI for the in app assistant.
- flutter_screenutil for responsive sizing and google_fonts for typography.
- intl and flutter_localizations with generated ARB files for English and Vietnamese.
- fl_chart for analytics charts and lottie for animations.
- shared_preferences for local session storage.

## Architecture

The app follows the BLoC pattern to keep the user interface separate from business logic.

- `lib/screens` holds the UI, organized by feature area such as authentication, home, chat, subscription, management, and profile management, and further split by the customer, staff, and admin roles.
- `lib/blocs`, `lib/events`, and `lib/states` contain the BLoCs together with the events they consume and the states they emit.
- `lib/repositories` talk to the backend and Firebase, and are the only layer that performs network calls.
- `lib/models` define the data classes, including request and response payloads, each with JSON serialization.
- `lib/widgets` provide reusable UI building blocks such as buttons, form fields, dialogs, and app bars.
- `lib/helper` holds shared utilities, configuration, and helpers for shared preferences, localization, and realtime data.
- `lib/l10n` and `lib/generated` hold the localization sources and the generated localization code.

## Getting started

### Prerequisites
- The Flutter SDK, version 3.4.1 or newer, with a matching Dart SDK.
- A configured device or emulator for your target platform.
- A running instance of the backend API. The base URL is set in `lib/helper/config.dart`.
- A Firebase project if you want realtime chat and storage to work.

### Setup
1. Clone the repository.
2. Install dependencies:
   ```
   flutter pub get
   ```
3. Point the app at your backend by editing `apiRoot` in `lib/helper/config.dart`.
4. Add your own Firebase configuration. Copy `android/app/google-services.json.example` to `android/app/google-services.json` and fill in your own values. That real file is ignored by git so secrets are never committed.
5. Provide the API keys at build time. The app reads them from Dart defines rather than hardcoding them:
   ```
   flutter run \
     --dart-define=FIREBASE_API_KEY=your_firebase_api_key \
     --dart-define=GEMINI_API_KEY=your_gemini_api_key
   ```

## Testing

Automated tests live in the `test` folder and cover the core helpers, model serialization, error handling, and a sample widget. See `test/README.md` for the full list of cases.

Run all tests with:
```
flutter test
```

## Localization

The app ships with English and Vietnamese. Localization sources are the ARB files in `lib/l10n`. After editing them, regenerate the localization code with:
```
flutter pub run intl_utils:generate
```

## Configuration notes

- The API base URL is defined in `lib/helper/config.dart`.
- API keys are never hardcoded. The Firebase and Gemini keys are read from Dart defines (`FIREBASE_API_KEY` and `GEMINI_API_KEY`) that you pass at build time.
- The Android Firebase config file `android/app/google-services.json` is ignored by git. Use `android/app/google-services.json.example` as a template.
- Session data such as the signed in customer, staff, or admin is stored locally through shared_preferences.
